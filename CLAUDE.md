## Architecture Overview

This is **LifeLog** (life_log), a Flutter personal life-tracking app. Dart SDK ^3.10.4, Flutter 3.38.5 stable.

**5 tabs**: 工时 (Work Log), 支出 (Subscription), 项目 (Photo), 面板 (Statistics), 我的 (Profile).

### Layered architecture (MVC + Repository)

Every data module follows this exact pattern:

```
lib/modules/<name>/
  <name>_model.dart      # Isar @collection + domain logic extensions
  <name>_model.g.dart    # Generated by build_runner
  <name>_controller.dart # GetX controller (state + UI logic)
  <name>_repository.dart # Data access layer, mediates between DbService and SyncService
  <name>_view.dart       # Flutter widget (reads controller via Get.find())
  views/                 # Sub-pages where needed
```

**Views never call DbService or SyncService directly.** Controllers call Repository, which orchestrates local DB + cloud push. This is enforced since v1.1.6.

### Dependency injection & routing

- **GetX** (`get: ^4.6.6`) handles all DI, routing, and reactive state.
- `main()` initializes services synchronously via `Get.put()` / `Get.putAsync()`: `DbService`, `ThemeController`, `LogService`, `AuthService`, `SyncService`.
- `AppBinding` is a placeholder; each page registers its own bindings (`TabsBinding`, `LoginBinding`).
- Routes are defined in `MyApp.getPages`: `/` (TabsView) and `/login` (LoginView).

### Local database (Isar)

- **Isar** (`isar: ^3.1.0`) — on-device NoSQL database.
- Three collections: `WorkLog`, `Subscription`, `PhotoItem`.
- `DbService` is the single entry point for raw DB operations. All queries filter by `ownerUserId` and exclude soft-deleted records (`deletedAt == null`) by default.
- Each model has sync fields: `syncId` (UUID v4), `remoteVersion` (optimistic lock), `deletedAt` / `pendingDelete` (soft delete), `isDirty`, `remoteId`.

### Cloud sync (Supabase)

- **Supabase** handles auth (email/password) and cloud storage for work_logs and subscriptions.
- Sync is **Pull-first, then Push**: on startup and on auth state change, `SyncService.syncAll()` pulls remote changes (incremental via server time cursor, or full on first sync), then pushes local dirty records.
- Push uses optimistic locking: updates include `version` condition check; conflicts trigger a remote re-fetch to refresh the local `remoteVersion`.
- Deletes are soft (tombstone), purged locally only after confirmed cloud sync.
- `SyncService.newSyncId()` generates UUID v4 locally — records get a `syncId` at creation time (not dependent on remote ID).
- `_activeSync` future reuse + 2-second debounce prevents duplicate sync triggers.

### Design system

Located in `lib/common/theme/`:
- `app_theme.dart` — Material 3 light/dark ThemeData using `ColorScheme.fromSeed`.
- `app_semantic_colors.dart` — Extension-based semantic tokens (success, warning, muted, severity-based due status colors).
- `app_colors.dart` — Raw palette constants.
- `app_spacing.dart`, `app_radius.dart`, `app_motion.dart`, `app_elevations.dart` — Design token classes.
- `ThemeController` persists user preference (system/light/dark) to GetStorage.

Shared widgets in `lib/common/widgets/` are pure UI, stateless — no controller dependencies.

### Key services

| Service | Role |
|---------|------|
| `DbService` | Raw Isar CRUD, owner-stamping, soft-delete helpers, sync helpers |
| `AuthService` | Supabase auth wrapper (`signIn`, `signUp`, `signOut`, reactive `currentUser`) |
| `SyncService` | Pull-merge-push sync protocol, UUID generation, debounce |
| `LogService` | Structured logging wrapper (info/error/debug) |
| `BackupService` | JSON export/import; throws exceptions rather than showing UI alerts |
| `ThemeController` | Theme mode persistence and switching |

### Supabase migrations

Located in `supabase/migrations/`:
1. `20260426_sync_identity_version.sql` — adds `sync_id`, `version` columns with unique constraint
2. `20260426_soft_delete_sync.sql` — adds `deleted_at`, `expenses`, `transport` columns
3. `20260426_server_time_rpc.sql` — creates `get_server_time()` RPC function for server-authoritative time cursors

### CI/CD

`.github/workflows/build.yml` triggers on `v*` tags: builds release APK (split per ABI) and publishes to GitHub Releases. Secrets required: `KEYSTORE_BASE64`, `STORE_PASSWORD`, `KEY_PASSWORD`, `KEY_ALIAS`.

### Internationalization

The app is Chinese-language UI. Lunar calendar support via `lunar: ^1.7.8`. Date formatting initialized with `initializeDateFormatting('zh_CN')`.

### Code generation

Isar models use `@collection` annotation. After editing any model (`*_model.dart`), run `dart run build_runner build --delete-conflicting-outputs` to regenerate `.g.dart` files.
